## フォルダ構成（リポジトリ）
- repo-root/
  - bookmarks/                # ブックマークエクスポート（1ファイル=1デバイス）
    - device-{id}.json
  - snapshots/                # スナップショット履歴（自動生成）
    - snapshot-{timestamp}.zip
  - metadata/
    - devices.json            # 登録デバイス一覧・公開鍵等
    - sync-state.json         # 最終同期情報
  - .gitignore
  - README.md

## デバイス保存ファイルフォーマット（device-{id}.json）
- JSONリスト形式。主なフィールド：
  - device_id
  - last_sync_iso8601
  - bookmarks: [{ id, url, title, tags[], folders[], created_at, updated_at, visit_count, source_device_id }]
  - version (incremental integer)

## DBスキーマ（ローカル SQLite — 省略可能だが推奨）
- tables: bookmarks, tags, folders, bookmark_tags, bookmark_folders, devices, sync_logs
- bookmarks: id (UUID), url, title, created_at, updated_at, visit_count, source_device_id, hash, deleted (soft)

## 同期ワークフロー（GitHubを中継）
1. 初回セットアップ：ユーザーが専用リポジトリを作成し、アクセストークン（スコープ: repo)をアプリに登録。ローカルは device-{id}.json を作成してコミット&push。
2. プルフェーズ：アプリは GitHub API でリポジトリをフェッチ（git pull 相当：GET contents or git refs/trees）。ローカルとリポジトリの device-*.json を比較。
3. マージフェーズ（ローカルで処理）：
   - 各デバイスファイルを読み込み、bookmark hash（正規化URL + title）でマージ案を作成。
   - コンフリクトはルールに従い自動解決（下記参照）またはユーザー承認キューに入れる。
4. プッシュフェーズ：ローカルで統合後の device-{id}.json と sync-state.json を更新してコミットし、リポジトリへ push（GitHub APIで create/update file）。
5. スナップショット：重大変更時に snapshots/に zip をアップロード（オプション）。

## コンフリクト解決ルール
- 同一bookmark id（UUID）かつ内容差分：
  - 更新日時 newer-wins（ただし visit_count は合算）。
- URLが同一だがタイトルやタグが異なる：
  - タイトルは最長非空を優先、タグはUnion。
- 同一URLで複数エントリ（重複）：
  - マージして1件に統合、元の source_device_id を履歴に保持。
- 同タイムスタンプの編集（競合）：
  - 自動でマージできない項目は user-review キューに入れる。Zedアシスタントで表示して解決可能。
- 削除の扱い：
  - ソフトデリートフラグを使い、他デバイスで同URLが存在すれば復元するか確認。

## Git操作の実装案（APIベース）
- 使用API: GitHub REST Contents API (GET /repos/{owner}/{repo}/contents/{path}) と Create/Update File (PUT).
- 手順:
  - 取得：GET contents for bookmarks/
  - 比較：ダウンロードdevice files, compute diffs
  - 更新：PUT for changed files with base64 content + commit message
- レート制限・並列注意。

## セキュリティと認証
- トークンはローカルのみ保存（OSシークレットストア推奨）。リポジトリはプライベート必須。
- 機密データ対策：ブックマークに含まれる可能性のある認証情報はクレデンシャルスキャナで警告。
- オプション: 各ファイルをローカルでAES-GCMで暗号化し、リポジトリには暗号化ファイルを保存（ユーザーがパスフレーズ保持）。
- 最低限のOAuthスコープを要求（repo: contentsのみ）。

## パフォーマンスと大規模データ対策
- device-{id}.json はデバイス単位で分割し、変更差分のみをPUT。
- 大量の小さなコミットを避けるためバッチ更新。
- ローカルはインデックス化（hash, url）。

## Zedアシスタント連携プロンプト例
- タスク例: 「未分類のブックマークを3つのタグに分類して」
- プロンプト雛形:
  - "以下のJSON配列のbookmarksをトピック別に3グループに分類して、各bookmarkに推奨タグ（最大3）を返して下さい。出力はJSONで: [{id, recommended_tags:[...]}]."
- コンフリクト解決時プロンプト:

CloudFlare workerでも可能か？

GPT-5 mini:
可能です。Cloudflare Workers を同期バックエンドとして使う設計上のポイントを短くまとめます。

主要方針
- GitHub リポジトリを直接操作する代わりに、Cloudflare Worker を「中継API」として配置し、Worker が GitHub API とやり取りする。もしくは Worker をリポジトリ代替のシンクポイント（KV/Pages/Workers KV / Durable Objects）として使う。

推奨アーキテクチャ（簡潔）
1. Worker as proxy/mediator（推奨、既存リポジトリ継続）
   - 各クライアントは直接 GitHub API を叩かず Worker に対して PUT/GET を行う。
   - Worker が認証トークンを安全に扱い、GitHub Contents API とやり取りする（ファイル取得・更新）。
   - 利点：クライアント側でトークン管理不要、レート制御・バッチ化・監査が容易。
2. Worker as primary sync store（リポジトリ不要）
   - Worker + Durable Objects / R2 / KV にブックマークを保存し、Worker がバージョン管理と履歴を提供。
   - 利点：低レイテンシ、細粒度の更新、より低コストな小規模運用。
   - 欠点：ユーザーがGitHubで履歴管理したい場合は不一致。永続ストレージコスト検討要。

実装上の注意点
- 認証：GitHubトークンをWorkerに置く場合はアカウントのトークン管理ポリシーを明確化（推奨はユーザー個別トークンをクライアントで取得し、短期トークンをWorkerに渡す方式か、OAuthフローをWorker経由で実行する）。Worker自体に長期トークンを置くとセキュリティリスク。
- レート制限：Workerでリクエスト集約・バッチ化・リトライを実装してGitHubレート制限を和らげる。
- 一貫性：Durable Objects を用いるとデバイス単位・bookmark単位で排他制御が可能。複数同時更新のコンフリクト解消が容易。
- ストレージ選択：
  - Workers KV：読み取り高速だが整合性は最終的。大きなオブジェクトや履歴管理には不向き。
  - Durable Objects：強い整合性・排他制御に適する（推奨で同期ロジックを担う）。
  - R2：大きなスナップショット/zip格納に適。
- セキュリティ：クライアント認証は短期JWTや署名付きリクエスト、CORS、TLS必須。秘匿情報は暗号化保管。
- コスト：Durable Objects と R2/KV の使用量により料金が発生するので、大規模ユーザーは見積もりを。

同期ワークフロー例（Worker as mediator）
1. クライアント → Worker: "push delta"（変更のみ）認証付き。
2. Worker: ローカルルールでマージ判定・コンフリクト検出。必要に応じて GitHub Contents API を呼び出して最新を取得。
3. Worker: マージ結果を GitHub にコミット（バッチ化）。更新結果を Clients に返す。
4. Worker は snapshots を R2 に保存し、sync-state を Durable Object に保持。

コンフリクト解決推奨
- Durable Object で bookmark-id ごとのロックを取得して更新（optimistic + last-write-wins with merge rules）。
- 自動解決しない場合は user-review キューを生成し、Zed アシスタントで提示。

運用上のメリット
- クライアントが GitHub トークンを持たなくてよい（トークン漏洩リスク低減）。
- レート制御やログ、分析、暗号化を中央で行える。
- 将来、GitHub を外して Cloud-only モードに移行しやすい。

短い推奨
- 小規模で既存GitHubワークフローを残したいなら「Worker as proxy/mediator」。
- 高い一貫性・低レイテンシを優先するなら「Worker + Durable Objects（主ストア）」。

必要なら、Worker-as-mediator の具体的な API 仕様（エンドポイント、リクエスト/レスポンス例）、Durable Object マッピング、または OAuth フロー図を作成しますか？
